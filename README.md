#  Twelve-factor app

在现代，软件通常以服务形式提供：称为网页应用 ，或软件即服务 。十二因素应用是一种构建软件即服务应用的方法论
本文是Twelve-factor app的摘要结果，原文地址：https://github.com/twelve-factor/twelve-factor.git

1. 12-Factor 应用必须严格遵循**一份基准代码对应一个应用**的原则。若存在多个应用共享代码，应将其拆分为独立类库。该应用可以基于这份统一的基准代码进行**多份部署**（如生产、预发布、本地开发），各部署可使用不同版本，但核心代码源必须一致。

2. 12-Factor 应用必须通过依赖声明清单明确列出所有依赖，并借助依赖隔离工具确保不隐式调用系统级库；同时，也不应隐式依赖系统工具，如需使用应打包进应用，以确保环境一致性

3. 12-Factor 应用要求代码与配置严格分离，将配置存储在环境变量中。环境变量按部署独立设置、粒度细小且不分组，既能轻松区分不同部署，又能彻底避免将敏感信息误签入代码库，确保代码库随时可安全开源。

4. 12-Factor 应用将后端服务视为附加资源，对本地和第三方服务一视同仁。应用只需通过修改配置中的 URL 或服务证书，就能在不改动代码的情况下实现服务的互换、加载和卸载，使部署与资源保持松耦合。

5. 12-Factor 应用严格分离构建、发布、运行三个阶段。构建阶段产出不可变的代码包，发布阶段将其与配置绑定生成唯一发布版本，运行阶段则在执行环境中启动该版本。任何变更都必须创建新的发布版本，且禁止在运行时直接修改代码。

6. 12-Factor应用必须以一个或多个无状态、无共享的进程运行，任何持久化数据都应存储在后端服务中，避免使用内存或文件系统进行依赖单进程的缓存，尤其不能使用粘性session。

7. 12-Factor 应用通过自我加载并绑定端口来直接对外提供服务，不依赖外部服务器容器，且该端口可服务于任意协议，甚至让应用本身成为其他应用的后端服务。

8. 12-Factor应用将进程视为一等公民，借鉴Unix守护进程模型，通过多样化的**进程类型**（如Web、Worker）来分担工作负载。其核心扩展策略依赖于进程的**无共享、水平分区**特性，以此实现简单稳妥的并发，并由操作系统进程管理器统一管理，而非应用自身守护。

9. 12-Factor应用的进程必须是**易处理**的，即能**快速启动**以增强敏捷性与健壮性，并在收到终止信号时**优雅终止**（网络进程停止监听并完成当前请求，Worker进程将任务退回队列）；同时，架构设计需遵循 **Crash-only design** 理念，确保进程在面对突然死亡时依然保持健壮。

10. 12-Factor 应用通过缩小**时间差异**（小时/分钟级部署）、**人员差异**（开发人员参与运维）和**工具差异**（环境高度一致），来确保**开发环境与线上环境等价**。关键原则是坚决反对在不同环境间使用不同的后端服务，所有环境必须使用**相同类型和版本**的后端服务，以消除因服务差异导致的不兼容，从而保障持续部署的顺畅。

11. 12-Factor 应用将**日志视为事件流**，每个进程只需将日志无缓冲地写入 **stdout**，完全不应自己存储或管理日志文件。在线上环境中，由运行环境负责截获并汇总所有输出流，统一路由至日志索引、分析系统或存档系统进行处理。

12. 12-Factor 应用的管理任务（如数据迁移、REPL控制台、一次性脚本）应作为一次性进程，在与常驻进程完全相同的环境中运行。它们必须基于相同的代码、配置和发布版本，并使用一致的依赖隔离技术，随应用代码一同发布以避免同步问题。
